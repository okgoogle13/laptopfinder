## Context

The Perplexity Workflow Manager (PWM) serves the laptopfinder eBay AU sniper pipeline. This spec adds 5 new Deep Research workflows that close gaps in the existing 13-workflow catalogue and feed the sniper's offline config at zero runtime API cost.

Core design constraint: the sniper (`ebay_sniper.py`) is zero-LLM. All intelligence comes from offline config — `config/static_reference_layer.json`, `prompts/search_queries.txt`, `data/seen_sniper_items.json`. Every new workflow must produce static artefacts consumed offline; zero further API calls once written.

---

## Gaps in existing catalogue

1. **No batch price floor refresh** — `lf-price-baseline` is per-GPU; 10+ calls to refresh all `target_gpus` entries.
2. **No watch_list graduation monitoring** — 4 entries (RTX 5000 Ada Mobile, RTX PRO 5000 Blackwell Mobile, GB200, B200) hold MONITOR status indefinitely; no workflow checks `graduation_condition`.
3. **No query expansion / blind-spot detection** — `prompts/search_queries.txt` has 39 queries; no structured process to find coverage gaps as new GPU models enter the AU market.
4. **No exclusion regex evolution** — `data_integrity.exclusion_regex` is static; no workflow discovers new AU junk listing patterns from corpus evidence.
5. **No clearance seller intelligence** — `clearance_sellers` and `watched_sellers = []` never updated; no workflow covers current stock or new AU refurb sellers.

### pwm lf-floor-sync

```
Tier: Deep Research
Purpose: Refresh observed_au_price_min_aud / observed_au_price_max_aud across all target_gpus
         tiers in one batch query, replacing 10+ individual lf-price-baseline calls.
Gap it fills: lf-price-baseline is per-GPU; no batch floor refresh exists; stale floors cause
              Strategy B to miss local bargains or fire on overpriced stock.
Sniper feed: target_gpus.<gpu>.observed_au_price_min_aud (Strategy B local price gate).
When to run: Monthly, or when any target_gpus entry has evidence_type = Inferred_global.
Reduces Gemini calls: none (sniper is zero-LLM; floors are offline config).

Inputs:
  - config/static_reference_layer.json (full target_gpus block with current price floors)
  - data/ebay/corpus.jsonl (recent eBay AU listing corpus for evidence)

Outputs:
  - data/pwm/lf-floor-sync/price_report.md — AU secondary-market price analysis across all
    target GPU tiers; citation-rich; human-readable
  - data/pwm/lf-floor-sync/price_patches.json — array of:
    {gpu, old_min_aud, new_min_aud, old_max_aud, new_max_aud, evidence_summary, confidence}
    Zero further API calls required to consume.

Steps:
  1. [free] Extract current price floors from SRL:
     make pwm-floor-sync-prep
     → paste output as context for the Deep Research prompt
  2. [deep] Perplexity Deep Research query:
     "Current AU eBay sold and listed prices for: RTX 3080 16GB laptop, RTX 3080 Ti 16GB
     laptop, RTX 4090 laptop, RTX 5080 laptop, RTX 5090 24GB laptop, RTX A4500 laptop,
     RTX 5000 Ada laptop, MacBook Pro M1 Max 64GB, MacBook Pro M2 Max 64GB, Strix Halo
     laptop — AUD sold prices, condition Used/Refurbished, eBay AU secondary market,
     July 2026. Return price band (p10–p90) per tier."
     → produces data/pwm/lf-floor-sync/price_report.md + price_patches.json draft

Done criterion: data/pwm/lf-floor-sync/price_patches.json exists and contains ≥ 8
               non-null {gpu, new_min_aud} entries.
               Validate: make pwm-floor-sync-check

PWM command: pwm lf-floor-sync
```

### pwm lf-watch-grad

```
Tier: Deep Research (+optional council-3)
Purpose: Check AU market availability for all watch_list graduation_conditions in one pass,
         producing a human-reviewable GRADUATE / HOLD / DEFER verdict per entry.
Gap it fills: Four watch_list entries hold MONITOR status indefinitely; no workflow checks
              graduation_condition; graduated GPUs never reach target_gpus and the sniper's
              Strategy A keyword set never expands.
Sniper feed: watch_list → target_gpus (via human promotion after review). Once promoted, the
             GPU name becomes a Strategy A flagship sweep keyword.
When to run: Monthly, or when Blackwell mobile enterprise leasing or RTX PRO 5000 AU retail
             news appears.
Reduces Gemini calls: none.

Inputs:
  - config/static_reference_layer.json (watch_list block — 4 entries with graduation_condition)

Outputs:
  - data/pwm/lf-watch-grad/graduation_report.md — per-entry status: condition met / not met,
    evidence links, GRADUATE / HOLD / DEFER recommendation
  - data/pwm/lf-watch-grad/graduation_candidates.json — array of:
    {name, graduation_condition, condition_met: bool, evidence_url,
     proposed_srl_patch: {vram_gb, platform_class, budget_band, observed_au_price_min_aud,
     observed_au_price_max_aud, evidence_type, market_verdict}}
    null proposed_srl_patch if condition not met

Steps:
  1. [free] Print watch_list graduation_condition strings from SRL:
     make pwm-watch-grad-prep
  2. [deep] Perplexity Deep Research query:
     "AU eBay, FB Marketplace, Gumtree, Grays Online — are any of these available in
     Australia as of July 2026: (1) RTX 5000 Ada Mobile laptop used domestic listings;
     (2) RTX PRO 5000 Blackwell Mobile laptop AU retail or used; (3) GB200 mobile or
     workstation AU availability; (4) B200 mobile or workstation AU availability.
     For each: provide evidence URL, price if found, confidence level."
     → produces graduation_report.md + graduation_candidates.json draft
  3. [optional, council-3] Only if condition_met = true for any entry AND proposed price
     band is ambiguous — council call to validate observed_au_price_min_aud before
     human commits SRL patch.

Done criterion: data/pwm/lf-watch-grad/graduation_report.md exists and contains a
               GRADUATE / HOLD / DEFER verdict for all 4 watch_list entries.
               Validate: make pwm-watch-grad-check

PWM command: pwm lf-watch-grad
```

### pwm lf-query-expand

```
Tier: Deep Research
Purpose: Identify coverage blind spots in prompts/search_queries.txt and propose new AU
         Browse API query strings for recently released or emerging GPU/chassis models.
Gap it fills: No workflow discovers new search coverage gaps; as Blackwell mobile, new
              Strix Halo derivatives, and post-Ada workstation GPUs enter the AU market,
              the query library stagnates without systematic expansion.
Sniper feed: prompts/search_queries.txt (batch runner — hunt.py / ebay_hunter.py).
             Indirect sniper effect: new queries surface GPU models that may be promoted to
             target_gpus, expanding Strategy A keywords.
When to run: Monthly, or on confirmed AU retail availability of a major new GPU launch.
Reduces Gemini calls: ebay_hunter enrichment (higher-signal queries reduce enrichment noise).

Inputs:
  - prompts/search_queries.txt (current 39 queries, with section comments)
  - config/static_reference_layer.json (target_gpus keys, watch_list names,
    uma_platforms.chip_patterns)
  - output/decisions/latest_decisions.json (what the pipeline is currently finding)

Outputs:
  - data/pwm/lf-query-expand/query_gaps.md — identified blind spots with rationale
    and Browse API constraint notes (no Boolean operators in q= parameter)
  - data/pwm/lf-query-expand/proposed_queries.txt — new query strings formatted
    identically to existing search_queries.txt sections, ready to paste in

Steps:
  1. [free] Dump current query list (non-comment lines) + SRL target GPU names:
     make pwm-query-expand-prep
  2. [deep] Perplexity Deep Research query:
     "AU eBay laptop GPU market July 2026: which high-VRAM (≥12GB) laptop GPU SKUs are
     available in Australia that are NOT in this query list: [paste list]. Focus on:
     Blackwell laptop GPUs with ≥16GB VRAM, new Strix Halo chassis, AMD workstation
     mobile GPUs with ROCm support, RTX PRO Blackwell mobile workstation. For each gap:
     proposed eBay keyword string (plain keywords only, no Boolean operators), AU
     availability evidence, VRAM capacity."
     → produces query_gaps.md + proposed_queries.txt draft

Done criterion: data/pwm/lf-query-expand/proposed_queries.txt exists and contains
               ≥ 3 new query strings not present in current prompts/search_queries.txt.
               Validate: make pwm-query-expand-check

PWM command: pwm lf-query-expand
```

### pwm lf-exclusion-tune

```
Tier: Deep Research (+optional council-3 with --strict)
Purpose: Discover new AU junk listing language patterns for data_integrity.exclusion_regex
         by analysing the eBay AU corpus and researching salvage/parts-only listing
         conventions on the AU secondary laptop market.
Gap it fills: exclusion_regex is static; no workflow evolves it as new junk patterns emerge.
              lf-risk-calibrator (Council) handles ambiguous complete-unit listings — this
              workflow targets the pre-filter firewall for clearly parts-only/salvage
              listings only.
Sniper feed: data_integrity.exclusion_regex (direct — Stage 2 grounding firewall +
             sniper zero-junk filter both check this pattern).
When to run: Per false-positive cluster (when sniper surfaces known-junk listings
             escaping the current regex), or at sprint start as a maintenance pass.
Reduces Gemini calls: run_stage2 (fewer junk listings reach grounding stage).

Inputs:
  - config/static_reference_layer.json (current data_integrity.exclusion_regex value)
  - data/ebay/corpus.jsonl (field confirmed: "title" — eBay corpus listing titles)

Outputs:
  - data/pwm/lf-exclusion-tune/junk_patterns.md — proposed new exclusion terms with
    AU eBay listing examples and false-negative risk notes
  - data/pwm/lf-exclusion-tune/exclusion_patch.json — structured:
    {current_regex, proposed_additions: [{pattern, rationale, example_fragment,
     false_negative_risk: low|medium|high}]}

Steps:
  1. [free] Extract current exclusion_regex and 50 recent corpus titles (field: "title"):
     make pwm-exclusion-tune-prep
     → paste both as context for the Deep Research prompt
  2. [deep] Perplexity Deep Research query:
     "AU eBay laptop listings July 2026: common phrases used in salvage, parts-only,
     and no-GPU chassis listings that would NOT be caught by this exclusion regex:
     [paste current regex]. Focus on: Australian seller phrasing conventions,
     BIOS-locked / board-only / GPU-damaged patterns, listings where item is a shell
     but the title includes a GPU model name. Propose new regex alternatives."
     → produces junk_patterns.md + exclusion_patch.json draft
  3. [optional, council-3] Only if any proposed_addition has false_negative_risk = high
     — adversarial council call to check whether the new pattern would block legitimate
     GPU listings (e.g., "no charger" might catch valid hardware-only bundles).
     Trigger with: pwm lf-exclusion-tune --strict

Done criterion: data/pwm/lf-exclusion-tune/exclusion_patch.json exists and
               proposed_additions array contains ≥ 2 patterns with false_negative_risk
               assessed as low or medium.
               Validate: make pwm-exclusion-tune-check

PWM command: pwm lf-exclusion-tune [--strict]
```

### pwm lf-seller-intel

```
Tier: Deep Research
Purpose: Research current AU eBay stock for existing clearance_sellers and identify new AU
         refurb/clearance sellers worth adding to clearance_sellers or watched_sellers.
Gap it fills: clearance_sellers never updated; watched_sellers = [] never populated;
              lf-vendor-audit covers risk/reputation — not current stock or new seller
              discovery. lf-alert-forge generates a watcher daemon but requires knowing
              which sellers to point it at; this workflow produces that intelligence.
              lf-alert-forge and ebay_sniper.py are complementary (different data sources);
              this workflow serves both.
Sniper feed: clearance_sellers array (ebay_deals.py); watched_sellers array (sniper
             seller-scoped sweeps — currently empty; direct sniper impact when populated).
When to run: Monthly, or at sprint start before make live.
Reduces Gemini calls: none.

Inputs:
  - config/static_reference_layer.json (clearance_sellers, watched_sellers current values)
  - data/ebay/corpus.jsonl (field: "seller_username" — sellers already seen by pipeline)
  - output/decisions/latest_decisions.json (seller patterns in recent pipeline hits)

Outputs:
  - data/pwm/lf-seller-intel/seller_report.md — per-seller stock intelligence:
    current AU eBay listing summary, price positioning, seller activity level;
    plus emerging seller candidates section
  - data/pwm/lf-seller-intel/seller_patches.json — proposed:
    {
      "clearance_sellers": [...additions],
      "watched_sellers": [...additions],
      "rationale_by_seller": {
        "<username>": {
          "rationale": "...",
          "verified_ebay_au_url": "https://www.ebay.com.au/usr/<username>",
          "corpus_evidence": true|false
        }
      }
    }
    verified_ebay_au_url is REQUIRED for every proposed addition.
    corpus_evidence = true if seller_username appears in data/ebay/corpus.jsonl.
    Human verifies URLs before propagating to SRL.

Steps:
  1. [free] Print current sellers + corpus seller_username values (sellers already seen):
     make pwm-seller-intel-prep
  2. [deep] Perplexity Deep Research query:
     "AU eBay July 2026 — current laptop stock and pricing for these sellers: delloutletau,
     lenovoaustralia, asusrefurbishedau. Also: which other AU eBay sellers specialise in
     refurbished or clearance high-VRAM laptops (RTX 3080+, Apple Silicon 64GB+, Strix Halo)?
     Provide seller usernames, verified eBay AU profile URLs, typical price positioning,
     and volume. Note if any of these sellers appeared in the corpus seller list: [paste]."
     → produces seller_report.md + seller_patches.json draft

Done criterion: data/pwm/lf-seller-intel/seller_patches.json exists; contains rationale
               entries for all 3 existing clearance_sellers; every entry in the
               clearance_sellers and watched_sellers additions arrays has a non-null
               verified_ebay_au_url of the form "https://www.ebay.com.au/usr/<username>"
               and a corpus_evidence boolean. Human verifies URLs before SRL propagation.
               Validate: make pwm-seller-intel-check

PWM command: pwm lf-seller-intel
```

### Naming convention gaps

1. **Output dirs inconsistent**: existing Deep Research uses `research/<type>_<date>.md` + `data/<type>_<date>.json`; Labs uses `apps/lf_*/`; Council uses `planning/`. New workflows adopt `data/pwm/<workflow-name>/`. Recommend migrating existing catalogue to same pattern in a future docs pass (not this sprint).
2. **Date stamping**: existing Deep Research outputs use `_<date>` suffixes. New workflows omit date stamps at file level; version tracked by git commit. Cleaner for single-file-per-workflow.
3. **Done-check paths**: no done-check paths in existing catalogue. New manifest entries add them explicitly; recommend backfilling.
