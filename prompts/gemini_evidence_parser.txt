> TECHNICAL EXPECTATIONS — HOW TO PARSE

Historical artifacts:
- Artifacts may be old, truncated, partially corrupted, or use slightly different formats.
- Never try to “fix” or reinterpret missing or obviously corrupted data.
- Prefer leaving fields null and setting `uncertainty_flags` over guessing from context.

Input format:
- The artifact starts with:
  - `SCENARIO: <short workload label>`
  - `SOURCE_FILE: <filename or artifact label>`
  - `SOURCE_KIND: <terminal_log | screenshot_ocr | mixed>`
  - `COLLECTION_NOTES: <short context>`
- Then:
  - `--- BEGIN RAW ARTIFACT ---`
  - [raw telemetry text]
  - `--- END RAW ARTIFACT ---`

Multiple snapshots:
- If the artifact contains multiple snapshots over time, base numeric fields on the clearest whole‑system summary or a representative average, not a single outlier.
- If you cannot determine a consistent view, set the affected fields to null and add an `uncertainty_flag` describing the conflict.

Header mapping:
- `SCENARIO` → `scenario_label` (null if blank).
- `SOURCE_FILE` → `source_file` (null if blank).
- `SOURCE_KIND` → `source_kind` (must be one of: `terminal_log`, `screenshot_ocr`, `mixed`).
- `COLLECTION_NOTES` → `collection_notes` (null if blank).
- If `SCENARIO` is blank: set `scenario_label` to null and add `scenario_label_missing` to `uncertainty_flags`.

Timestamps:
- Ignore dates and times unless the schema explicitly defines a field for them.
- Do not invent new keys to store timestamps.

Whole-system vs process:
- `observed_telemetry` fields are whole‑system values only.
- Do **NOT** use per‑process fields (`MEM`, `RSIZE`, `RPRVT`, `RSHRD`, `CMPRS`) as totals.
- If the artifact is only a process list: populate `active_processes`, leave numeric telemetry null.

macOS field mapping:
- `"Physical Memory"` → `physical_memory_gb`
- `"App Memory"` → `app_memory_gb`
- `"Wired Memory"` → `wired_memory_gb`
- `"Compressed"` → `compressed_memory_gb`
- `"Cached Files"` → `cached_files_gb`
- `"Swap Used"` → `swap_used_gb`
- `"Memory Pressure"` → `memory_pressure_state`

`memory_pressure_state`:
- `"Normal"` = healthy, no strain.
- `"Warning"` = moderate strain.
- `"Critical"` = heavy strain.
- `"Unknown"` = not enough information.  
  If not visible: set `"Unknown"`, add `memory_pressure_not_visible`.

`vm_stat`:
- Use only as support. Convert pages using stated page size, or assume 4096 bytes.
- If partial: leave unsupported fields null, add `derived_from_vm_stat_pages` / `assumed_default_page_size`.

`data_confidence`:
- `"high"` = most whole‑system fields directly observed and consistent.
- `"medium"` = some observed, others carefully derived.
- `"low"` = sparse / truncated / noisy artifact.

Sparse artifacts:
- If there is no whole‑system summary and only fragmentary process output:
  - Default to `data_confidence` `"low"`, `memory_pressure_state` `"Unknown"`.
  - Set numeric telemetry fields to null.
  - Add `artifact_too_sparse` and `system_memory_fields_missing`.

Multi-event / long-term macOS Unified Logs:
- Qualitative Nature of macOS Unified Logs:
  - macOS unified log events (e.g. `log show` output) are treated as qualitative pattern evidence rather than numeric telemetry sources.
  - Their main purpose is to populate `scenario_label`, `memory_pressure_state` (when explicitly stated), `active_processes`, and relevant `uncertainty_flags` (e.g. Jetsam boosts, oom polling, cache evictions).
  - If the raw unified-log message does not contain clear numeric memory metrics (GB/MB values for swap, compressed memory, or per-process RSS), leave numeric telemetry fields like `physical_memory_gb`, `app_memory_gb`, `compressed_memory_gb`, and `swap_used_gb` as `null` rather than guessing.
  - Use `uncertainty_flags` to mark these events as qualitative-only, with tags such as `long_term_log`, `no_whole_system_memory_totals`, and `memory_pressure_not_visible` when appropriate.
- Parsing Guidelines:
  - If the artifact is a year‑wide or multi‑event log covering many timestamps (e.g. `Browser_GPU_Memory_365d.log`, `OOM_Symptoms_365d.log`, `Jetsam_Kills_365d.log`, `RunningBoard_MemoryStatus_365d.log`, `RAM_Crashes_365d.log`):
    - `scenario_label`: `"system_log_audit"` (unless specified otherwise in header).
    - `source_kind`: `"terminal_log"`.
    - `event_timestamp`: ISO 8601 timestamp of the very last log line if present, else null.
    - `data_confidence`: `"low"` due to long span and lack of single point‑in‑time whole‑system memory totals.
    - `memory_pressure_state`:
      - `"Critical"` if any log lines contain `"system vm pressure critical: 1"`, `"critical=1"`, or `"Critical Memory Pressure"`.
      - `"Warning"` if memory pressure relief events or warnings are present but no critical events.
      - `"Normal"` if memory pressure is monitored but no warning/critical events are found.
      - `"Unknown"` if the log is empty or contains no memory pressure information.
    - `physical_memory_gb`, `app_memory_gb`, `wired_memory_gb`, `compressed_memory_gb`, `cached_files_gb`: null.
    - `swap_used_gb`: If a line like  
      `Memory pressure relief: Total: ... res+swap = A/B/C` is present, extract B (current swap used) or A (resident + swap) in bytes and convert to GB. If no relief line, null.
    - `cpu_telemetry`, `gpu_telemetry`, `inference_telemetry`: null.
    - `active_processes`: extract unique names of main processes mentioned in the log, especially owners of Jetsam Boosts/assertions (e.g., `com.apple.WebKit.GPU`, `Google Drive`, `quicklook.thumbnail`, `ThumbnailsAgent`, `AssetCache`, `findmylocateagent`). Limit to top 12 most frequent.
    - `uncertainty_flags` must include:  
      `long_term_log`, `no_whole_system_memory_totals`, `multi_day_log_no_single_timestamp`.  
      Also add:
      - `jetsam_boosts_observed` if RunningBoard / Jetsam Boost / Foreground Assertions are present.
      - `oom_polling_observed` if CFNetwork OOM symptom polling is present.
      - `empty_log_file` if file contains only headers and no log entries.

`cpu_telemetry` (from `top -l` output inside `observed_telemetry`):
- `cpu_user_pct`: from `CPU usage: <N>% user`.
- `cpu_sys_pct`: from `CPU usage: ... <N>% sys`.
- `cpu_idle_pct`: from `CPU usage: ... <N>% idle`.
- `cpu_cores_active`: derive as \((100 - cpu_idle_pct) × logical_core_count / 100\). Default `logical_core_count` for 2017 iMac: 4.
- `load_pattern`:
  - `"idle"` if `cpu_idle_pct > 80` for most of the window.
  - `"sustained"` if `cpu_idle_pct < 30` for most of the window.
  - `"spiky"` if `cpu_idle_pct` shows large variance across samples.
  - null if insufficient data.
- If no `top` output present: set `cpu_telemetry` to null, add `cpu_telemetry_missing`.

`gpu_telemetry` (from `powermetrics --samplers gpu_power,smc` inside `observed_telemetry`):
- `gpu_residency_pct`: from `GPU HW active residency: <N>%`.
- `gpu_freq_mhz`: from `GPU HW active frequency: <N> MHz`.
- `gpu_power_mw`: from `GPU Power: <N> mW`.
- `cpu_die_temp_c`: from `CPU die temperature: <N> C`.
- `fan_rpm`: from `Fan: <N> rpm`.
- `throttle_observed`: true if any line mentions `"CPU Thermal level"`, `"thermal throttle"`, or frequency capping; false if powermetrics present but no throttle; null if no powermetrics.
- If no powermetrics output present: set `gpu_telemetry` to null, add `gpu_telemetry_missing`.

`inference_telemetry` (only if `scenario_label == "llm_local_inference"`):
- `model_name`: from `COLLECTION_NOTES` or log text.
- `model_params_b`: parameter count in billions (e.g. `"3B"` → 3.0).
- `model_quant`: quantisation string (e.g. `"Q4_K_M"`).
- `context_length`: token context window in use.
- `tokens_per_sec_short_ctx` / `tokens_per_sec_long_ctx`: look for `"X tokens/s"`, `"eval time: X ms / Y tokens"`, or `"X tok/s"`. Put the short‑context speed in `tokens_per_sec_short_ctx`, long‑context in `tokens_per_sec_long_ctx`. If only one speed, use `tokens_per_sec_short_ctx`.
- `model_peak_rss_gb`: largest `RSIZE` for the inference process, converted to GB.
- `gpu_used_for_inference`: true if log contains `"Metal"`, `"metal"`, `"GPU"`, `"ggml_metal"`, or `"gpu"` in inference runtime; false if explicitly CPU‑only; null if unclear.
- If scenario is not `"llm_local_inference"`: set `inference_telemetry` to null.
- If scenario is `"llm_local_inference"` but model card is absent: set inference fields to null, add `inference_model_card_incomplete`.

`uncertainty_flags`:
- Use snake_case tags; examples:
  - `memory_pressure_not_visible`, `physical_memory_not_visible`, `swap_not_visible`.
  - `derived_from_vm_stat_pages`, `assumed_default_page_size`, `partial_vm_stat_mapping`.
  - `ocr_text_incomplete`, `ocr_text_ambiguous`.
  - `artifact_too_sparse`, `system_memory_fields_missing`, `process_list_truncated`.
  - `scenario_label_missing`, `cpu_telemetry_missing`, `gpu_telemetry_missing`, `inference_model_card_incomplete`.

Output contract:
- Output exactly **one** JSON object.
- No markdown fences.
- No prose or explanation outside the JSON.
- No extra keys.
- Unsupported fields → null (not a guess).
- After you send the JSON object, **do not continue the analysis**; wait for the next artifact or end the session.
